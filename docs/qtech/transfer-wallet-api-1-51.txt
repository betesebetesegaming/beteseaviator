1 Introduction引言 
This document describes the Transfer Wallet integration endpoints, which the Operator needs to integrate 
to be able to perform money transfers between the player wallet in the Operator system and the player 
wallet in QT Platform.  
The document is applicable for Operators where the QT Casino wallet is being handled by the QT 
Platform.  
本文档描述了转移钱包集成终端，运营商需要集成终端以便能够在运营商系统中的玩家钱包和QT平台中的玩家钱包之间进
行资金转移。 
 
该文件适用于运营商的QT赌场钱包由QT平台处理。 
1.1 Audience读者 
This document is suitable for System/Solution Architects, Software Developers, Technical Project 
Managers and Technical Business Analysts. 
该文件适用于系统／方案架构师，软件开发，技术项目经理和技术业务分析师。

2 Overview概观 
 
The server-to-server communication between the Operator system and QT Platform is uni-directional 
(one way). All requests must be performed from the Operator system. 
A wallet transfer between the Operator system and QT Platform is being performed in two (2) steps, to 
be more failsafe and to allow cancellations or automatically rollbacks.  
The first step prepares the wallet transfer, which creates a PENDING transaction within the QT Platform. 
The second step completes or cancels the transfer.  
If a PENDING transfer has not being completed (or cancelled) within 10 seconds, the transfer will 
automatically be CANCELLED. A transaction which is in its final state, i.e. COMPLETED or CANCELLED, can 
never be changed. 
运营商系统和QT平台之间的服务器到服务器的通信是单向的（单程）。所有请求都必须从运营商系统进行。 
在运营商系统和QT平台之间的钱包转账需要两（2）个步骤执行，以更安全地并且允许取消或自动回滚。 
第一步准备钱包转账，在 QT 平台内创建一个 PENDING 交易。 第二步完成或取消转账。 
如果待定的转账在10秒内还没有被完成（或取消），此转账将自动被取消，这是它的最终交易状态，即完成或已取消，永远不
能改变。 
2.1 Formats格式 
The Transfer Wallet endpoints in QT Platform are being exposed as RESTful web services with JSON as 
the data-interchange format and with the character encoding UTF-8.  
The currency code within the QT Platform, i.e. in all requests and responses, is expressed as 3-letters 
code according to standard ISO 4217. For example “CNY”.  
The country code within the QT Platform is presented as 2-letter code according to standard ISO 3166-1 
alpha-2. For example “CN”. 
The language code with QT Platform is presented as a combination of the 2-letter language code (ISO 
639-1) and the 2-letter country code (ISO 3166-1 alpha-2). For example “zh_CN”. 
The date and time within the QT Platform is being presented in CST (China Standard Time) according to 
standard ISO 8601. The time is presented in “military time”, i.e. 24-hours clock. 
QT平台中的Transfer Wallet端点作为RESTful Web服务公开，使用JSON作为数据交换格式，并使用UTF-8编码。 
在QT平台中的货币代码，即在所有的请求和响应，符合标准ISO 4217例如“人民币”被表示为3字母代码 “CNY”。 
在QT平台中的国家代码是按照标准ISO3166- 1 alpha-2显示为2个字母的代码。例如，“CN”。 

与QT平台的语言代码是符合标准 ISO 639-1表现为2个字母的代码。例如，“zh_CN” 
符合标准 ISO 8601的时间显示在“军用时间”，即24小时时钟QT平台中的日期和时间被呈现在CST(中国标准时间）。 
 
Format格式 
YYYY-MM-DDThh:mm:ss 
Example案例 
2015-09-10T13:14:51 
 
2.2 Idempotency幂等 
All of the transfer wallet endpoints are idempotent, meaning same transfer can be retried in case of error 
but the retry will not result to new transaction in QT Platform. A retry will return the original response. 
所有转账钱包端点都是幂等的，这意味着在出现错误的情况下可以重试相同的交易，但重试不会导致 QT Platform 中的新交
易。 重试将返回原始响应。 
 
2.3 Player Creation 创建玩家 
There's no specific “Create Player” endpoint. A Player gets automatically created on the first fund transfer, as described 
in “Prepare a Transfer 准备一个转账” or when launching a game, as described in “Game Launcher游戏启动” 
没有具体的“创建玩家”端点。当玩家第一次转账的时候这个玩家账号就被自动创建了，在“准备转账”中有相关描述，或者当启
动一个游戏的时候，在“游戏启动”中有相关描述。 
 
2.4 Error Handling错误处理 
Sometimes errors can happen during a request. In case of an error, the response must be reflected both 
in the header (HTTP status code) and with a response body, containing an error code and message. All 
errors will return HTTP status code 4xx or 5xx. 
有时在请求期间可能发生错误。 在发生错误的情况下，该响应必须同时在头部（HTTP状态码）和响应主体（包含错误代码
与信息）中体现。 所有错误将返回HTTP状态码4xx或5xx。  
 
 
 
Each API definition will explain in detail what possible errors (http status, code and message) that can 
occur. 
每个API定义将会详细解释可能出现的错误（http status, code and message)。 
 

2.5 Free Round Promotions免费游戏局活动 
The QT platform supports two types of free round promotions; Fixed or Flexible.  
A Fixed promotion is where the operator decides a fixed number of rounds and amount for a certain 
player and promotion.  
A Flexible promotion is where the operator can automatically generate different round options for the 
player to choose from, based on the rewarded bonus amount. This option gives more flexibility for the 
player, to select a round option that best matches his/her bet strategy. 
QT 平台支持两种类型的免费游戏局：预设定的或者可调整的 
预设定的免费游戏局是指运营商为指定的玩家和活动，选择预定的游戏局数量和每局的投注额。 
可调整的免费游戏局是指运营商可以根据奖励的金额自动生成不同的游戏局选项供玩家选择。这个选项可以为玩家提供更
大的灵活性，选择最符合他/她的投注策略的游戏局选项。 
 
To create free round promotions, the operator can either use the campaign tool in Back Office or use our 
API, as explained in chapter “8.8 Create Free Round Promotion”.  
运营商可以使用QT后台的活动工具或者通过API来创建免费游戏局，在API文档8.8章节“CreateFreeRoundPromotion创建
免费游戏局 ” 有详细说明。 
2.5.1 Free Round Result 免费游戏局活动结果 
The result of a played free round promotion will be delivered in one or several game rounds, where each 
round will start with a 0-amount withdrawal transaction followed by one or several deposit transactions. 
The bonus type and the promotion code, which was given when creating the promotion, will be passed in 
every withdrawal and deposit transaction. 
For example – a player has been promoted with 2 free rounds with a bet amount of 20.00 CNY per round, 
with promotion code “123”. 
一个完成的免费游戏局活动的结果会作为一个游戏局或者分成多个游戏局来传递。每个游戏局会以一个金额为0的提款交
易开始，后面会有一个或多个存款交易。 
示例： 一个玩家得到2局免费游戏活动，每局游戏的投注金额为20元，免费游戏局活动代码为“123”

The status of a certain promotion can be displayed in the Back Office or be requested from the API, as described in “Get

A Free Round Promotion can be in one of the following states:

4 Transfer Wallet 
转账钱包 
The Transfer Wallet API describes five (6) endpoints; “Prepare Transfer”, “Complete Transfer”, “Cancel 
Transfer”, “Retrieve Transfer Details”, “Retrieve Player Balance” and “Promotion Status”. 
The QT Platform accepts virtual currencies as player wallet currency. The Operator can choose the game 
currency its player will use.  
The following are the supported virtual currencies: BTC (Bitcoin), ETH (Ethereum), XRP (Ripple), LTC 
(Litecoin), BCH (Bitcoin Cash), BNB (Binance Coin), BUSD (Binance USD). Since these currencies are 
volatile in nature, the conversion rates that QT Platform offer comes from a trusted third party exchange 
rates provider that we fetch on a regular basis. We will also be able to support custom fixed currencies 
which are based on FIAT currencies through this feature. 
QT 平台接受虚拟货币作为玩家钱包货币。运营商可以选择其玩家将使用的货币。运营商系统应识别并接受 
walletCurrency 和兑换率对象作为钱包操作的有效参数。
 
 
以下是支持的虚拟货币：BTC（比特币）、ETH（以太坊）、XRP（瑞波币）、LTC（莱特币）、BCH（比特币现金）、BNB（币安
币）、BUSD（币安美元）。由于这些货币本质上是不稳定的，QT 平台提供的兑换率来自我们定期获取的值得信 
 
 
Players will be able to use a virtual currency in the wallet and play the game in a common (ISO) 
currency, with a given conversion rate to convert the amounts in the transactions and requests to the 
player wallet.  This conversion rate should be accepted by the player and will be used for the whole game 
session.  
玩家将能够在钱包中使用虚拟货币并以通用 (ISO) 货币玩游戏，并以给定的转换率将交易金额和请求中的金额转换到玩家
钱包。这个转化率应该被玩家接受，并将用于整个游戏会话。 
4.1 Prepare a Transfer 准备一个转账 
The “Prepare Transfer” endpoint creates a PENDING transfer transaction within the QT Platform. The 
transfer type can either be a CREDIT, which indicates a deposit to the QT Wallet or a DEBIT, which 
indicates a withdrawal from the QT Wallet.  
“准备转账”端点在 QT 平台内创建一个 PENDING 转账交易。 转账类型可以是表示存款到 QT 钱包的 CREDIT，或是表示从 
QT 钱包提款的 DEBIT。 
 
If the transfer type is a DEBIT, the amount will be reserved from the player's balance in QT Wallet. If the 
transfer type is a CREDIT, the amount will be credited to the player's balance upon completion.  
如果转账类型是 DEBIT，所述量将从玩家 QT 钱包的余额里保留。如果转账类型是 CREDIT，所述量将在完成时记入玩家的
余额里。

sessions of players who have logged out of the system or in similar scenarios, such as players being idle

8.13  Tournament 锦标赛 
The Tournament API endpoints, which allow an authenticated Operator to retrieve tournament 
information and leaderboard rankings from the QT Platform 
锦标赛 API 的各项端点，经身份验证的运营方（Operator）可通过该 API 从 QT平台获取锦标赛信息和排行榜排名数据。 
The Tournament API provides two endpoints for retrieving tournament data: 
• Get Tournaments — Returns a paginated list of tournaments available to the authenticated operator. 
• Get Leaderboards — Returns leaderboard rankings for a specific tournament. 
All communication is server-to-server, over HTTPS, using JSON as the data-interchange format with 
UTF-8 character encoding. Date and time values are presented in ISO 8601 format. 
锦标赛 API 提供两个用于获取锦标赛数据的端点： 
• Get Tournaments — 返回经过身份验证的运营商可访问的锦标赛分页列表。 
• Get Leaderboards — 返回特定锦标赛的排行榜排名数据。 
所有通信均为服务器到服务器通信，通过 HTTPS 进行，使用 JSON 作为数据交换格式，并采用 UTF-8 字符编码。 
Tournaments are competitive events where players accumulate points based on win conditions (e.g., Best 
Round or Collect Points). Each tournament has an availability window defined by a start and end date, 
and may include single (overall) prizes, daily prizes, or both. 
锦标赛是玩家根据获胜条件（例如最佳回合或收集积分）累积积分的竞技赛事。每个锦标赛都有由开始和结束日期定义的可
用窗口，可能包括单一（总体）奖品、每日奖品或两者兼有。 
By default, the Get Tournaments endpoint returns only IN_PROGRESS tournaments (started before now, 
ending after now). OPEN (future) and COMPLETED tournaments can be retrieved with explicit date 
parameters. 
默认情况下，Get Tournaments 端点仅返回进行中（IN_PROGRESS）的锦标赛（即已开始且尚未结束的锦标赛）。未开始（
OPEN）和已结束（COMPLETED）的锦标赛可通过指定日期参数进行查询。 
 
8.13.1 Leaderboards / 排行榜 
Each tournament has one or more leaderboard groups depending on its availability type: 
• Single-only: One group with type SINGLE (group 0) 
• Daily-only: Multiple groups with type DAILY (group 1, 2, ...) 
• Single + Daily: One OVERALL group (group 0) + multiple DAILY groups 
Players outside the requesting operator's hierarchy have their operatorId and playerId masked (e.g., 
al****USD). 
每个锦标赛包含一个或多个排行榜分组，具体取决于其奖励结构： 
• Single-only（仅单一奖励） 
  - 仅包含 1 个分组（group 0） 
  - 类型：SINGLE 
• Daily-only（仅每日奖励） 
  - 包含多个分组（group 1、2、3......） 

  - 类型：DAILY 
• Single + Daily（单一 + 每日奖励） 
  - group 0：OVERALL（总排行榜） 
  - group 1、2......：DAILY（每日排行榜） 
超出当前请求运营商层级范围的玩家，其 operatorId 和 playerId 将被脱敏显示（例如 al****USD）。

8.13.4 Get Tournaments / 获取锦标赛 
Returns a paginated list of tournaments available to the authenticated operator. By default, only 
IN_PROGRESS tournaments are returned. OPEN (future) and COMPLETED tournaments can be retrieved 
with explicit date parameters. INCOMPLETE tournaments are always excluded. 
Results are sorted by startDateTime ascending. 
返回经过身份验证的运营方可访问的锦标赛分页列表。 
默认仅返回进行中（IN_PROGRESS）的锦标赛。 
可通过指定日期参数获取未开始（OPEN）和已结束（COMPLETED）的锦标赛。 
未完成（INCOMPLETE）的锦标赛始终不包含在返回结果中。 
查询结果按 startDateTime 升序排序。

8.13.8 Get Leaderboards / 获取排行榜 
Returns leaderboard rankings for the specified tournament. Rankings include player rank, operator, player 
ID, and points. 
Players outside the requesting operator's hierarchy have their operatorId and playerId masked (e.g., 
ot****er1, ja****ane). 
The response includes one or more leaderboard groups depending on the tournament's availability type: 
● Single-only: One group with type (group 0) 
● Daily-only: Multiple groups with type (group 1, 2, ...) 
● Single + Daily: One Overall group (group 0) + multiple Daily groups 
返回指定锦标赛的排行榜排名数据。排名信息包括玩家排名、运营商、玩家ID以及积分。 
超出当前请求运营商层级范围的玩家，其operatorId和playerId将被隐藏/掩码处理（例如 ot****er1、ja****ane）。 
根据赛事类型，排行榜将分为一个或多个组： 
●   单项赛：一个组（组 0） 
● 每日赛：多个组（组 1、2 等） 
● 单项赛 + 每日赛：一个总排名组（组 0）+ 多个每日排名组
